4096-RGB

una imagen que contenga todas las combinaciones en el esquema RGB producida por un “programa de ordenador”, puede ser «ARTE»? … puesta en una exposicion, cualquier mierda es arte

una nuevo pro(mesa|puesta) para revivir al intento de programador (artistico y conceptual) que pretendia ser. El problema parecia sencillo, parece ser fuente de nuevos y desconcertantes misterios matemagicians, perceptivos que tienen (o no) relacion con las representaciones digitales de aspectos de mundo real, como el c-o-l-o-u-r

primero lo fundamental, cuantos colores se supone pueden representarse con tres valores enteros sin signo de 8 bits?

(2^8)^3

# 2 por los valores que pueder almacenar cada bit (0 y 1)
# ^8 por los 8 bits
# ^3 de cada valor, es decir R (2^8) * G (2^8) * B (2^8)

# que es igual a
== 16,777,216

ahora de que dimenciones debe ser una imagen que contenga los dieciseis como siete millones de pixeles del rango RGB? … a saber, por mera casualidad probe con 2048 * 2048 = 4,194,304. Luego con 4096 * 4096 y… 16,777,216, puhaa! la pura magia de la casualidad binaria… 8*3 == 12… 2^12 == 4096… La verdad, no tengo idea de por que esto funciona

pues ya esta, a establecer los pixeles de un rectangulo de 4096*4096, pero como demonios hacer esto?

la primer idea en aparecer, fue dentro de un bucle establecer un indice a cero e iterar sumando 1 por ciclo hasta llegar a 16,777,216, algo asi

for i := uint32(0); i < 16777216; i++ {
  // hacer cosas con i
}

para cortar el entero y optener los colores, se recortaria el indice en sus tres octetos de bits menos significativos

[00000000][00000000][00000000][00000000] // i := uint32
             AZUL      VERDE     ROJO

esto usando “movimientos a nivel de bits” y “truncamiento por cast”, por supueto

r, g, b = uint8(i), uint8(i >> 8), uint8(i >> 16)

aunque enfrentado directamente el problemo y tras varias iteraciones el codigo adquirio esta forma estilizada y mas o menos conpacta

package main

import( "image"; "image/color"; "image/png"; "os" )

func main() {
  const width, height = 4096, 4096
  i, r, g, b := uint32(0), uint8(0), uint8(0), uint8(0)
  img := image.NewRGBA(image.Rect(0, 0, width, height))

  for   py := 0; py < height; py++ {
    for px := 0; px <  width; px++ {
      r, g, b = uint8(i), uint8(i >> 8), uint8(i >> 16)
      img.Set(px, py, color.RGBA{ r, g, b, 255 })
      i++
    }
  }

  png.Encode(os.Stdout, img);
}

unas constantes que establecen el ancho de la imagen. Variables para el indice, campos r, g, b y “la imagen”

dos bucles for para moverse de arriba a abajo de izquierda a derecha de la imagen estableciendo pixel a pixel y al final codificando la imagen en la salida estandar

para ejecutar esto, copiar el codigo y luego con el go ejecutar asi

> go run rgb.go > arte-RGB.png

se produce una imagen de 176KiB a-s-i ->

interesante la distribucion del color, similar al que produce una oblea de silicio…

mientras tanto, me preguntaba como demonios organizar de mejor forma la produccion de tonalidades y de ser posible patrones de lo mas extravagantes y pseudo fractales, o algo similar. (Nota: averiguar como…)

en otro breve lapsus, parecia mejor idea organizar la grabacion de pixeles con respeto a su suma rgb, es decir, de 0 a 767 (256 + 256 + 256 == 768)

(nota segunda: es facil liarse y por familiaridad pensar en cosas como 255*3 o 255^3, no caigais en el error mas viejo del mundo, el desplazamiento en uno… como yo)

me encontraba divagando si un programa que generara todos los valores RGB (cada color con 256 valores disponibles y almacenado cada uno en enteros sin signos de 8 bits, es decir 0-255)

package main

import( "image";"image/color";"image/png";"os" )

func main() {
  const width, height = 4096, 4096
  var colors [768][]color.RGBA
  var r, g, b uint8

  for i, sum := uint32(0), 0; i < 16777216; i++ {
    r, g, b = uint8(i), uint8(i >> 8), uint8(i >> 16)
    sum = int(r) + int(g) + int(b)
    colors[sum] = append( colors[sum], color.RGBA{ r, g, b, 255 } )
  }

  img := image.NewRGBA( image.Rect(0, 0, width, height) )
  for i, y, x := 0, 0, 0; i < 766; i++ {
    for _, rgb := range colors[i] {
      if x == width { x = 0; y++ }
      img.Set(x, y, rgb )
      x++
    }
  }

  png.Encode(os.Stdout, img);
}

esta ves se utiliza un arreglo de slices (arreglo redimencionable). Contiene 768 (256*3) slices, donde la posicion 0 es la suma (RGB) de 0 + 0 + 0 y 765 (255*3) 255 + 255 + 255… (este dezplazamiento en uno es muy confuso). El resto de valores intermedios, vedria a ser una combianacion de todos los valores para tal suma

para establecer los pixeles basta con recorrer el arreglo “trozo a trozo” (ha, ha, ha, que humor tan sofisticado)

el resultado, con apariencia mas aleatoria, parece tener cierto orden aplicando un acercamiento (gThumb)

por si las moscas, revizando con el programa para manipular imagenes de GNOME (alias “gimp”) parace que en efecto los 253^3 colores distintos estan hay

(CLICK IZQUIERDO -> colores -> info -> analisis del cubo de color)

y para terminar otra vercion de ordenamiento, pero esta ocacion (tambien) de arriba a bajo, pero el recorrido de izquierda a derecha continua por la derecha y luego sigue por la izquierda “serpenteando”

mientras tanto pensaba como precindir del (tramposo y poco eficiente) arreglo de slices y ocurrioseme que habiendo un indice de 1 a n, se troza para optener de esto tres valores (r, g, y b) y a su vez a estos se les aplican todas las posibles rotaciones, por ejemplo

indice  R   G   B      rotaciones
 ...   ... ... ...       5  0  0
  5     5   0   0        0  5  0
 ...   ... ... ...       0  0  5

y luego, o mejor planteado al indice inicial encontrar sus variaciones

indice  R   G   B   /->  variaciones   /-> rotaciones
 ...   ... ... ... /       5  0  0  --/      5  0  0
  5     5   0   0 /        4  1  0           0  5  0
 ...   ... ... ...         3  1  1           0  0  5
                           3  2  0
                           2  2  1
